# 贡献指南（CONTRIBUTING）

## 提交流程

1. 从 `main` 拉取最新代码并新建分支：`feature/xxx` 或 `fix/xxx`。
2. 完成改动后自测通过，再提交 PR（或合并请求）。
3. PR 描述需要写清：改动目的、影响范围、回滚方式（如有）。

## 代码与目录约定

1. 小程序代码集中在 `campus_treehole/`。
2. 云函数代码集中在 `campus_treehole/cloudfunctions/`，函数名与目录名保持一致。
3. 不在无关目录混入业务逻辑；静态素材按页面或模块分组管理。

## 提交信息建议

- `feat:` 新功能
- `fix:` 缺陷修复
- `docs:` 文档更新
- `chore:` 工程或配置维护

示例：`feat: 增加校园便利盒商品详情页筛选能力`

## 安全与配置

1. 禁止提交密钥、令牌、账号、数据库连接串等敏感信息。
2. `project.private.config.json` 仅本地使用，不应入库。
3. 需要共享配置时，优先更新示例配置文件（如 `*.example.json`）。

## PR 最低检查清单

1. 页面/云函数改动是否与需求一致。
2. 是否影响现有主流程（发布、登录、下单/互动等）。
3. 是否同步更新了必要文档与配置说明。
